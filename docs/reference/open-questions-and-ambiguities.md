# Decision Register And Deferred Details

This file is the first Rust-slice decision register. It replaces the earlier generic uncertainty list with resolved decisions, deferred implementation details, and non-questions that future coding agents must not reopen casually.

## Resolved Decisions For First Rust Slice

| Area | Decision | Contract impact | Owner |
| --- | --- | --- | --- |
| Crate boundaries | `agent-sdk-core` owns domain contracts, public IDs, run/runtime API, event bus, cursors, journal ports, provider/tool/policy ports, fakes, and contract tests. Toolkits, isolation adapters, extension SDK, OTel exporters, workflow orchestration, and host adapters are optional crates or host layers. | Core must compile without product/toolkit/runtime-adapter features. | 00/01 |
| MVP primitive kernel | The first Rust slice is profile-gated: P0 proves one fake-provider text run with `Agent`, `AgentRuntime`, `RunRequest`, `RunHandle`, `RunResult`, per-run effective `RuntimePackage`, first-slice `CapabilitySpec`, `ArtifactRef`/`ContentRef`, context admission/projection, shared effect intent/result, event frames, run journal, entity/source/destination refs, policy refs, and typed IDs. P1 adds typed output over the same loop. P2 adds tools/approval side effects. Isolation, extensions, subagents, realtime, stream rules, full tool packs, telemetry exporters, and global event archive replay remain reserved feature layers unless their profile activates them. | Phase goals must not require the full feature taxonomy to pass P0 or P1. | 00/01/02/03 |
| Phase-first workstreams | Numbered folders under `docs/workstreams/[0-9][0-9]-*/` define dependency phases. Every goal file directly inside a numbered folder is parallel-safe with its siblings. `_roles/` docs define ownership and writable scope; they are not launch targets. | Future `/goal` prompts should point to one numbered phase goal doc and obey that goal's writable list plus its owner role doc. | 00 |
| Phase delivery and reviewer gate | Multi-goal phases launch one subagent per direct goal file, consolidate goal packets into a phase-local `_phase/phase-exit-report.md`, run phase-level audits, then require a dedicated reviewer-agent PASS before the next numbered phase starts. Single-goal stitching phases run serially but still use reviewer gates. | Phase README exit gates must not be checked until goal packets, audits, proposal decisions, phase report, and reviewer verdict all pass. | 00 |
| Capability variant discipline | `CapabilitySpec` is limited to callable/discoverable capabilities with projection/executor/policy/sidecar refs. Provider route, output contracts, output sinks, hooks, guardrails, telemetry, lifecycle, and isolation are package fields or typed sidecars unless exposed as callable/discoverable capabilities. Each new variant beyond the MVP profile must appear in the runtime-package reserved-variant readiness table with typed sidecar contract, owner role, fingerprint fields, emitted events, journal records, and acceptance tests before it can be emitted or executed. | Runtime-package contract and review matrix must enforce sidecar ownership and reject inactive reserved variants outside test-only fake profiles. | 00/01 |
| Runtime event listening | Core SDK exposes first-class lifecycle subscriptions: `runtime.subscribe_all(cursor)`, `runtime.subscribe_run(run_id, cursor)`, `runtime.subscribe_agent(agent_id, cursor)`, and `runtime.subscribe_events(filter, cursor)`. `RunHandle::stream_from(cursor)` is the run-scoped convenience form, not the only event API. | `AgentRuntime`, `AgentEventBus`, `RunHandle`, and reconnect contracts must share cursor semantics. | 01/02 |
| Event filter shape | `EventFilter` matches cheap indexed envelope fields: run IDs, agent IDs, turn IDs, event families, event kinds, source, destination, subject/related entity refs, correlation keys, tags, privacy classes, and delivery semantics. | `agent_id`, entity refs, correlation keys, and tags are envelope/index fields, not payload-only fields. | 02 |
| Event hot path | Live filtering is envelope/index based. It must not parse JSON payloads, copy raw content, query content stores, or scan journals on the live fanout path. Compiled filters may use hash sets, bitsets, and precomputed match plans. | `CompiledEventFilter` and acceptance tests must prove payload-free matching. | 02/09 |
| Event delivery frame | Event streams yield `EventFrame { event, cursor, archive_cursor, overflow }`. The frame carries the delivered-frame `EventCursor`, optional `ArchiveCursor` for indexed archive replay, and optional `EventOverflowNotice` with policy, dropped count, gap cursors, optional repair cursor, terminal-preserved flag, and reason. | `AgentEventStream` item type is a frame, not a bare payload. | 01/02 |
| Cursor distinction | `EventCursor` is a live or recently buffered event-stream cursor. `JournalCursor` is the per-run durable replay cursor. `ArchiveCursor` is the optional cross-run indexed archive cursor. Cursor scope compatibility is exact: all resumes all, run resumes the same run, agent resumes the same agent, and filter resumes only the same filter fingerprint. Run-scoped replay can be guaranteed from the run journal through core. Cross-run/all-event durable replay requires a host-provided `EventArchive` or indexed journal view; core must not pretend a global durable query exists without that port. | Reconnect and replay docs must use both cursor types precisely. | 01/02 |
| Live vs durable fanout | Live event fanout and durable journal append are separate. Live subscribers cannot create durable facts, and slow subscribers cannot block journal append or agent execution. Terminal catch-up can use journal cursors when available. | Event bus and journal tests must cover overflow, gap notice, and replay catch-up. | 02 |
| Payload privacy | Default event subscribers receive `EnvelopeOnly` or `RedactedSummary`. Raw/full payload access is opt-in and only delivered when content-capture policy allows it. | Full payload requests must fail closed or downgrade visibly when policy denies content access. | 02/09 |
| Event-driven orchestration boundary | Core makes orchestration easy by exposing filters, cursors, lineage, and idempotent `start_run`. DAG engines, barriers, "wait for N agents," scheduling, compensation, workflow UI, and durable trigger state stay in optional workflow crates or hosts. | Examples may show a host/optional crate counting terminal events and starting a follow-up run, but core must not own that engine. | 00/01/02/10 |
| ID taxonomy | Public newtypes exist for `RunId`, `AgentId`, `TurnId`, `AttemptId`, `EventId`, `EventCursor`, `JournalCursor`, `TraceId`, `SpanId`, `EntityRef`, content/artifact refs, effect IDs, source refs, destination refs, policy refs, correlation keys, privacy classes, retention classes, and trust classes. | No ambiguous raw strings in public durable or cross-boundary DTOs. | 00/01 |
| Event payload freeze | Freeze envelope fields, event family/kind strings, privacy semantics, delivery semantics, and minimum DTO fields now. Exhaustive per-kind payload details can be finished through emitted-kind fixtures. | New/renamed families or kinds require compatibility notes and golden fixtures. | 02/09 |
| Journal atomicity | Append shared effect intent before execution and shared effect result after execution. If intent append fails, do not execute. If result append fails after execution, enter recovery and block further non-idempotent effects until reconciled. | Tool, approval, isolation, output dispatch, provider calls, memory writes, extension actions, child starts, and structured output records must use or map to intent/result pairs where side effects exist. | 02/04/06/11 |
| Runtime package fingerprint | MVP fingerprints come from the resolved per-run effective package. They include provider route, projected schemas, executable refs when active, `RunRequest.output_contract` normalized into package sidecars for P1, output sink/delivery policy when used, policy refs, typed sidecars, catalog snapshots that affect activation, and any fake tool used by tests. Reserved feature-layer fields join the fingerprint only when their typed sidecar contract is implemented. Host extension manifest fields such as browser-safe exports, package compatibility, trust state, install/runtime metadata, and action permissions are excluded unless represented as SDK-facing capability/policy refs. | Fingerprint generation must be deterministic and covered by golden tests for the active profile. | 00/01/04 |
| Lifecycle hooks | Hooks are first-class reserved lifecycle contracts: code-first `agent.on(...)` and declarative config both lower into `HookSpec` sidecars/executor refs in `RuntimePackage`. Hook inputs default to content refs/redacted summaries, observe hooks are nonblocking by default, and security hooks cannot fail open. Extension hooks are just hook executors behind optional extension/host bridges. | `hook-lifecycle-contract.md`, runtime package fingerprints, event kinds, and journal records must stay aligned. | 00/01/02/08 |
| Child lifecycle ownership | `RuntimePackage` declares default/allowed child lifecycle policy refs. `RunRequest` may select or tighten one before start. Manual cancel cascades to agent-owned child artifacts by default. Normal completion cannot leave implicit orphan work; explicit detach requires intent, acknowledgement, durable owner/reclaim records, and policy. | Run, tool, isolation, subagent, journal, and event contracts must share `RunChildLifecyclePolicy` semantics. | 00/01/02/04/06/07 |
| Provider projection | Providers receive `ContextProjection`, not internal messages. Unsafe/private metadata is stripped by default; projected metadata requires explicit policy and audit fields. | Projection audit fields are required journal/event data. | 03/09 |
| Context and memory ownership | `ArtifactRef`/`ContentRef`, `ContextContribution`, `ContextItem`, `ContextProjection`, optional `MemoryPort`, and compaction are owned by the context/structured-output role with stitching approval for shared primitive names. Memory, tool results, skills, host input, remote channels, subagents, and compaction return candidates; `ContextAssembler` decides admission under policy and budget. Context cannot become a universal SDK bag or shadow run state owner. | Context/memory contract owns projection audit, selection decisions, memory events, and `ContextRecord` payload variants for retrieval, selection, projection, compaction, memory write intent, and memory write result. | 03 |
| Structured output schema registry | Support both inline schemas and registry/content references. Registry refs are preferred for durable and replayable workflows; typed-model ergonomics lower into the same schema-first contract. | `agent.run_typed::<T>` derives or looks up a schema, validates locally, repairs within policy, then constructs `T`. | 03 |
| Stream reasoning channels | Stream rules can match public model output, provider-exposed reasoning summaries, tool output, or realtime transcript channels only when those channels are policy-visible. Hidden chain-of-thought is not matchable. | Stream-rule channel enums must encode privacy and provider visibility. | 05/09 |
| Tool reversibility | SDK records lineage, effect summaries, and inverse candidates when available. It never promises automatic undo or product-level revert. | Reversible metadata is advisory and must be policy-visible. | 04 |
| Isolation fallback | Requested isolation fails closed unless the `RuntimePackage` and host policy explicitly accept a class, capability, or trust-vector downgrade/fallback. Downgrades are evented and journaled with missing capability/trust fields. | Unsupported-host behavior must not silently run locally. | 06 |
| Extension packaging | Node ESM support uses normal `node_modules` package resolution and verified public subpaths. `NODE_PATH` fallback remains unsupported unless separately verified later. | Packaging smoke tests must cover ESM import, subpaths, and browser-safe helpers. | 08 |
| Output delivery ownership | `DestinationRef`, `OutputSink`, output delivery policy, dedupe, and intent/result records are owned by the output-delivery role. Product channel UX, credentials, notification copy, and durable channel stores stay host-owned. | Output delivery cannot bypass run journal, privacy policy, or dedupe semantics. | 11/00 |
| Host adapter timing | Product-specific host adapter implementation waits until core event, run, journal, runtime-package, and fake-adapter tests are stable. Generic scenarios remain coverage, not core authority. | Host docs may reveal missing primitives but cannot bypass core contract review. | 10/00 |

## Non-Questions

- The SDK is standalone and product-neutral.
- Host scenarios are coverage, not SDK core.
- Product self-improvement UX stays outside the SDK.
- Runtime-wide event listening is a core primitive.
- Event filtering is envelope/index based, not payload parsing.
- Raw prompt/model/tool/file content capture is off by default.
- Live event fanout is not durable storage.
- A run journal is durable for that run; a global durable event query requires an explicit archive/index port.
- Core recovery repairs derived internal views only; external compensation and product-facing repair workflows stay outside core.
- Ergonomic helpers lower into canonical contracts and cannot bypass validation, policy, journal, event, telemetry, lineage, or redaction.
- Feature goals layer on the primitive kernel instead of creating parallel event streams, journals, package registries, policy paths, or side-effect paths.
